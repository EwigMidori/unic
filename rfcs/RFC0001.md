# RFC 0001: 宏观架构设计与模块划分

*   **状态**: Draft (草案)
*   **版本**: 1.0.0
*   **依赖**: RFC 0000

---

## 1. 核心目标 (Core Objectives)
*   **高并发**: 支撑大量 Agent 同时进行异步决策。
*   **高扩展性**: 支持通过增加 Crate 快速适配不同的 LLM 或增加新的系统工具。
*   **容错隔离**: 尽管运行在同一进程，但确保一个 Agent 的崩溃不会导致内核调度器崩溃（Panic Handling）。

## 2. 工作空间结构 (Workspace Layout)

建议采用 Rust Workspace 模式，将系统解耦为以下物理 Crate：

```text
unicos/
├── Cargo.toml              # Workspace 定义
├── crates/
│   ├── unicos-core/        # 内核调度器 (Orchestrator)
│   ├── unicos-bus/         # 消息总线与事件模型 (Pub/Sub)
│   ├── unicos-agent/       # Agent 状态机与执行逻辑
│   ├── unicos-llm/         # LLM 抽象层与适配器
│   ├── unicos-tools/       # 系统工具箱 (Bash, FS, Network)
│   └── unicos-common/      # 共享模型、错误处理与类型定义
├── src/
│   └── main.rs             # 应用程序入口 (CLI & STDIO 绑定)
└── bin/                    # 编译产物存放处
```

## 3. 核心 Crate 职责定义

### 3.1 `unicos-bus` (通信中枢)
*   **职责**: 定义 `Message` 结构和 `Topic` 路由。
*   **实现**: 基于 `tokio::sync::broadcast`。
*   **关键数据结构**:
    *   `Event`: 包含 `Message`, `Command`, `SystemNotification`。
    *   `Envelope`: 包装消息及其元数据（发送者 ID、时间戳、目标 Topic）。

### 3.2 `unicos-core` (内核调度器)
*   **职责**: 管理 Agent 的生命周期，维护全局路由表，初始化系统服务。
*   **功能**:
    *   `AgentRegistry`: 存储当前活动的协程句柄（JoinHandle）。
    *   `FsWatcher`: 监听 `/etc/unicos/unics/`。
    *   `GodLogger`: 将所有消息持久化到 `/var/lib/unicos/god.log`。

### 3.3 `unicos-agent` (Agent 运行时)
*   **职责**: 定义单个 Unic 的循环逻辑。
*   **功能**:
    *   `DecisionEngine`: 核心逻辑，通过 LLM 判定“是否响应”。
    *   `ContextManager`: 负责管理滑动窗口记忆及从文件读取 `soul.md`。

### 3.4 `unicos-llm` (解耦适配层)
*   **职责**: 抹平不同模型 API 的差异。
*   **抽象**:
    *   `trait LlmProvider`: 定义 `complete()`, `decide()`, `stream()` 等方法。
    *   实现 `OpenAiProvider`, `OllamaProvider` 等。

### 3.5 `unicos-tools` (系统调用层)
*   **职责**: 提供 Agent 与外部世界交互的能力。
*   **原子操作**:
    *   `BashExecutor`: 执行 shell 命令。
    *   `FileSystemOperator`: 读写任意路径（无沙箱）。

## 4. 核心交互流程 (Data Flow)

### 4.1 消息触发流
1.  **输入**: 用户在 STDIN 输入，或 Agent A 调用 `speak` 工具。
2.  **路由**: `unicos-bus` 接收到 `Envelope`，广播至对应 `Topic`。
3.  **接收**: 所有订阅了该 `Topic` 的 `unicos-agent` 实例并行接收到消息。
4.  **决策**: 每个 Agent 调用 `unicos-llm` 进行 `should_i_respond?` 判定。
5.  **输出**: 若判定为是，Agent 启动 `generate_action` 流程，最终可能产出新的 `Message` 回流至总线。

### 4.2 生命周期控制流
1.  **用户操作**: `rm -rf /etc/unicos/unics/Alice`。
2.  **感知**: `unicos-core` 的 `notify` 模块捕获 `Remove` 事件。
3.  **清理**: 内核通过 `AgentRegistry` 找到 Alice 的 `AbortHandle` 并强制结束协程，释放内存。

## 5. 关键 Trait 定义 (Interfaces)

```rust
// unicos-llm/src/lib.rs
#[async_trait]
pub trait LlmProvider: Send + Sync {
    async fn generate(&self, prompt: Vec<ChatMessage>) -> Result<String, UnicError>;
    async fn should_respond(&self, context: Perception) -> bool;
}

// unicos-tools/src/lib.rs
#[async_trait]
pub trait SystemTool: Send + Sync {
    fn name(&self) -> String;
    async fn call(&self, args: Value) -> Result<Value, ToolError>;
}
```

## 6. 错误处理与容错 (Error Handling)

*   **内核守护**: `unicos-core` 使用 `tokio::select!` 监听各 Agent 协程。若某个 Agent 崩溃（Panic），内核通过 `catch_unwind` 或检测 `JoinError` 来记录错误，但不影响其他 Agent。
*   **自毁恢复**: 当关键路径被删除导致内核无法运行时，由 Docker 的 `restart: always` 策略配合只读镜像层进行系统重建。
