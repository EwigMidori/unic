# RFC 0000: UnicOS 架构设计规范

*   **状态**: Draft (草案)
*   **版本**: 1.0.0
*   **作者**: UnicOS Team
*   **日期**: 2024-05-23

---

## 1. 摘要 (Summary)
UnicOS 是一个基于 Rust 和 Tokio 协程构建的多 Agent 模拟与协作系统。它将 AI Agent（称为 "Unic"）视为系统内的轻量级进程。该系统采用“肉身入局”设计，即内核与 Agent 共享同一个执行环境。Agent 具备高度自主权，能够通过工具操作文件系统、联网、编写并运行代码。用户作为具有 `sudo` 权限的特殊节点，通过 STDIO 或简单的 IM 界面与系统交互。

## 2. 设计哲学 (Philosophy)
*   **去中心化自主**: 每个 Unic 独立决定何时发言、如何行动。
*   **物理真实性**: 对话、记忆和性格皆为文件。Agent 可以互相读取甚至篡改对方的这些数据。
*   **允许破坏**: 允许 Agent 执行具有破坏性的指令（如删除系统文件），依靠外部 Docker 镜像进行定期备份和灾难恢复。
*   **解耦**: 核心二进制程序、Agent 逻辑、运行数据与外部 LLM 提供商完全分离。

## 3. 目录与文件布局 (Filesystem Layout)

系统严格遵循 Linux 规范，确保数据、逻辑与二进制产物的分离：

| 路径 | 类型 | 描述 |
| :--- | :--- | :--- |
| `/usr/local/bin/unicos` | **只读** | Rust 编译后的内核二进制产物 |
| `/etc/unicos/config.toml` | **只读/管理** | 全局配置（LLM API Keys, 默认模型等） |
| `/etc/unicos/unics/{id}/` | **读写** | **灵魂区**: 存储 Agent 的 `soul.md` (System Prompt) 和 `config.json` |
| `/var/lib/unicos/god.log` | **追加** | **上帝日志**: 存储所有 Topic 的客观对话流数据 |
| `/home/{id}/` | **读写** | **工作空间**: Agent 的主目录，包含其私有记忆文件和代码产物 |

## 4. 核心架构组件 (Core Components)

### 4.1 消息总线 (The Bus)
*   基于 **Tokio Broadcast Channel** 实现。
*   支持 **Topic** 机制（如 `#general`, `#project_x`）。
*   每个 Unic 协程可以订阅多个 Topic，监听流经该频道的所有消息。

### 4.2 记忆模型 (Memory Model)
*   **客观现实 (Objective Reality)**: `god.log` 记录所有消息，Agent 无法直接访问完整版。
*   **感官投影 (Perception Window)**: 内核在唤醒 Agent 时，仅截取最近的 $N$ 条消息注入上下文。
*   **自主持久化 (Autonomous Persistence)**: Agent 若需记住窗口外的消息，必须主动将其写入 `/home/{id}/` 下的文件。

### 4.3 执行模型 (Execution Loop)
每个 Unic 运行在一个独立的 `tokio::spawn` 循环中：
1.  **感知**: 接收总线消息。
2.  **决策**: 调用 LLM 判定是否需要响应（基于性格、上下文及当前话题）。
3.  **行动**: 若决定响应，可选择 `Speak` (发送文本) 或 `Tool_Call` (如 `bash_run`, `fs_read`, `net_request`)。
4.  **反馈**: 工具执行结果返回总线，再次触发感知循环。

## 5. 管理与交互 (Management & Interaction)

### 5.1 用户角色
用户标识为 `[User: sudo]`，通过 STDIO 接入：
*   **参与者模式**: 普通输入即在当前频道发言。
*   **管理员模式**: 使用 `/` 前缀发送指令（如 `/spawn`, `/kill`, `/join`, `/sleep`）。

### 5.2 生命周期管理 (Lifecycle)
*   **增加**: 在 `/etc/unicos/unics/` 新建目录触发内核 `inotify` 自动创建协程。
*   **删除**: 删除对应目录或发送 `/kill` 指令，内核 `abort` 对应协程。
*   **休眠**: 修改 Agent 状态字段，内核暂停其对总线的监听，但保留磁盘状态。

## 6. 技术栈 (Technical Stack)
*   **内核语言**: Rust
*   **异步运行时**: Tokio
*   **文件监听**: `notify` crate
*   **LLM 适配**: 自研 Provider 层，支持 OpenAI 协议及本地 Ollama。
*   **容器化**: Docker (负责定时镜像快照及进程崩溃后的自动重启)。

## 7. 安全与风险 (Risk & Mitigation)
*   **自毁**: 从提示词禁止自毁，但不做实际的硬性限制。
*   **恢复**: 系统状态的持久性依赖于 Docker 的外部卷挂载（Volume）和镜像回滚机制。内核本身不负责数据的强一致性保证。
*   **碰撞**: 多个 Agent 抢话时，通过总线的顺序到达进行自然竞争，不强制线性排队。
